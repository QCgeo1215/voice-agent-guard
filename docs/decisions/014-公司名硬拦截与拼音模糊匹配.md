# 014 公司名：硬拦截白名单 + 拼音模糊匹配

## 背景 / 约束
- 6/21 真机联调：司机报「全新科技公司」(不在园区 11 家租户名单)，后端 `register_visitor` 正确返回 `success:false`「没查到来访单位」，于是**没登记、没推送企业微信**；Agent 还反复确认公司 4 次、最后自己补了句「这就通知保安」(谎报)。
- 暴露的根本冲突：用户第一需求「企业微信必须收到通知」 vs 本次要求「公司白名单要硬限制」——**硬拦截下，名单外公司必然没通知**。
- 用户拍板：① 维持**硬拦截**(只有名单内能登记)；② 后端公司匹配要**真正的模糊匹配**，别只靠手列别名，用中文/拼音相似度算。

## 候选方案（名单外公司怎么处理）
1. **登记并标注 flag**：名单外也登记 + 推送，消息标「⚠️不在名单，请核实」。通知必达，限制变可见提醒。
2. **硬拦截（本次采用）**：名单外拒绝、不登记；Agent 诚实说「不在登记名单，转人工」，绝不谎报已通知。最贴近「封闭园区只放行已知租户」的安防语义。
3. 名单外也照常登记（无任何限制）：等于没有白名单，被用户否决（「不然怎么限制」）。

## 候选方案（怎么提升匹配召回）
- A. **继续手工堆别名**：每遇到一个听错变体就往 `companies.json` 加。脆、累、覆盖不全。
- B. **拼音相似度（本次采用）**：中文转无声调拼音串，与别名拼音算 `SequenceMatcher`，与字面相似取 `max`。一招覆盖大批同音错字（鲸鱼/金鱼/精于→jingyu；陈兴/晨星→chenxing），且**真不同的公司拼音也不同，不会误命中**，不破坏硬拦截。
- C. 上 LLM 重排：已有开关 `COMPANY_RESOLVE_USE_LLM`，但吃延迟预算、要 key，作为可选增强而非主链路。

## 最终选择
名单策略选 **2（硬拦截）**；匹配增强选 **B（拼音相似度）**，叠加在原有「别名精确 + 字面相似」之上，阈值不变（自动接受 `≥0.82 且 margin≥0.18`，建议确认带 `0.75–0.82`）。

## 实现
- `company_registry.py`：
  - 加 `pypinyin==0.55.0`；`_pinyin()` 转无声调拼音串(中文按音、英文/拼音原样)。
  - `_alias_pinyin()`（`lru_cache`）预算「别名→拼音」一次；`reload_company_registry()` 同步清这层缓存。
  - `_rank_candidates()` 每个别名取 `max(字面 ratio, 拼音 ratio)` 聚合到标准名。
  - `normalize_company()` 逻辑不变：无可信匹配仍抛 `UnknownCompanyError`（硬拦截）。
- Prompt（`vapi/system_prompt.md`）配合硬拦截，修两个真 bug：
  - **铁律：register_visitor 没返回成功，绝不说「已通知保安/已登记」**(根治谎报)。
  - 公司不在名单：念候选一次；司机说都不是/坚持名单外公司 → 不再反复确认同一个名字，直说「不在登记名单，帮您转人工」后结束(根治死循环)。
- 测试：新增 `test_company_homophone_via_pinyin`——`陈兴物流→晨星物流`(字面仅 ~0.75，靠拼音命中)、`全新科技公司` 仍抛 `UnknownCompanyError`(白名单回归)。本地 `pytest` 29 绿。

## 理由
- 硬拦截符合封闭园区「只放行已知租户」的安防直觉，比 flag 更能体现「限制」的价值；通知必达由「让司机报名单内公司 / 转人工」保证，而非牺牲限制。
- 拼音相似度用算法替代手工别名，覆盖 STT 主要错误模式(同音)，且对硬拦截零副作用(异音公司不会被拉近)。

## 已知代价 / 可演进
- 名单外公司确实进不来 → 走人工，是设计取舍而非缺陷；演示需用名单内 11 家公司。
- 拼音忽略声调，极端情况下不同声调同拼音可能拉近——但 11 家租户拼音两两差异大 + margin 校验兜底，实测无误命中。
- 可演进：需要时再开 `COMPANY_RESOLVE_USE_LLM` 做候选重排；或接真实园区企业目录替换 demo 白名单。
