# 008 手机端入口方案（二维码 Web Call）

## 背景 / 约束
- 理想流程是「停车场入口张贴电话号码，司机拨打电话」。
- 电话号码调研已确认：个人短期内拿不到合规 +86 AI 呼入号；HR 也反馈国外号 / 网页 demo 可接受，电话不是重点。
- 现有 Vapi dashboard 的 **Talk to Assistant** 只适合开发测试，不像真实业务入口。
- 目标：不重写语音链路、不碰复杂电话资质，把 demo 入口从电脑升级到手机端。

## 候选方案
1. **继续用 Vapi dashboard Web Call**：零开发，但观感像内部测试工具，不适合作为演示入口。
2. **二维码手机 Web Call（本次采用）**：后端提供 `/call` 手机页和 `/qr` 二维码页；司机扫码后用手机麦克风和同一个 Vapi Assistant 对话。
3. **真实电话号码 / SIP**：最贴近真实场景，但需要 DID/SIP 采购与配置，且国内 +86 仍受资质限制，当前时间成本高。

## 最终选择
方案 2：二维码手机 Web Call。

实现：
- `GET /call`：移动端访客入口，嵌入 Vapi Web Call widget。
- `GET /qr`：按当前请求域名生成 `/call` 二维码；cloudflared 地址变化时重新打开即可。
- 环境变量：
  - `VAPI_PUBLIC_KEY`
  - `VAPI_ASSISTANT_ID`

## 理由
- 司机仍用手机完成语音登记，业务体验接近真实入口。
- 复用现有 Vapi Assistant、`lookup_visitor`、`register_visitor`、后端、微信推送，风险低。
- HTTPS 由 cloudflared 提供，满足手机浏览器麦克风权限要求。
- 可清楚说明：demo 入口是 Web Call；生产可替换为 SIP / 电话号码，业务核心不变。

## 已知代价 / 可演进
- 严格说不是「拨打电话号码」，是手机浏览器语音通话。
- 微信内置浏览器 / 部分 iOS 场景可能限制麦克风；页面已提示使用系统浏览器打开。
- Vapi Public Key / Assistant ID 需要在 `.env` 配置，且 Vapi prompt / tool URL 仍需控制台同步。
- 生产演进：换成正式域名 + 固定二维码；或接入 SIP/DID，把入口从 `/call` 替换为电话号码。
