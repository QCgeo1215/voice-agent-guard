# 003 回访识别（老访客）

> ⚠️ 升级说明（6/18）：本决策选的方案 3「事后欢迎回来」已被**决策 007** 升级为方案 2「对话内 lookup 主动识别」。
> 升级原因：把"开口主动确认式回访"列为理想态（"张师傅您好，今天是不是和上周二一样来蓝色鲸鱼送货？"），优先匹配它。
> 本文保留，记录当时为何先选简单方案；新结论见 `007-对话内回访识别.md`。

## 背景 / 约束
- 可选增强：识别老访客、提升体验。
- demo 是 Web Call，**没有主叫号（caller ID）**，无法"一接通就认出"；只能用对话中拿到的**手机号**作回访键。
- 后端按手机号查历史是本地 SQLite 查询，几 ms，不影响 25s 延迟预算。

## 候选方案（识别深度）
1. **只推送标注**：登记时查历史，命中在微信推送加标注。司机端无感、不省询问。
2. **对话中识别 + 复用少问**：新增 `lookup_visitor` 工具，agent 拿到手机号先查、命中复用历史字段。最出彩，但**对话内多一次工具往返**（多一个 LLM + 网络回合，吃延迟），且 LLM 要正确编排两个工具，稳定性风险高。
3. **回访感知 + 欢迎回来（不改收集流程）**：正常收集 4 项，后端登记时查历史，命中则成功话术加"欢迎回来" + 推送标注。

## 最终选择
方案 3。
- 后端：`_register_core` 在 insert 前用 `count_visitors(phone)` 取历史次数、`find_latest_by_phone(phone)` 取上次记录；命中则话术用 `SPEECH_SUCCESS_REVISIT`，推送 `_format_message` 加【回访】块。
- prompt：结束语改为"念工具返回的话"，让后端单点控制话术（成功/失败/回访都由后端给）。
- 主叫号识别（生产最优）写入演进路线，受 demo 无线路限制不演示。

## 理由
- 延迟拆解：吃 25s 的是"对话往返次数"和模型/网络，不是后端查库。方案 3 不新增对话往返、只多两次本地查库（几 ms），延迟几乎无变化。
- 方案 2 的"少问"虽亮眼，但为一个可选增强引入对话编排与延迟风险，性价比低（与查询 Agent"避免过度设计"的取舍一致）。

## 已知代价 / 可演进
- 仍问全 4 项，没省询问（只有体验 + 标注）。
- 回访键是手机号：同号不同人、或换号，会误判 / 漏判。
- 生产可上主叫号识别 + 对话复用（方案 2），实现"接通即认出 + 少问"。
