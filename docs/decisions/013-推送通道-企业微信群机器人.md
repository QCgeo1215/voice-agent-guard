# 013 推送通道：云上改用企业微信群机器人

## 背景 / 约束
- 6/21 真机联调发现：访客登记成功、入库正常，但**保安一直收不到微信**。
- 后端日志（`/register_visitor` 返回的 `push_detail`）显示：ECS（AWS 新加坡）连 Server酱 `sctapi.ftqq.com` **connect 超时 8s**，而同一时刻**本机直连 Server酱只要 1.9s**。
- 约束：① 仍要「微信家族」通知，保住「微信通知保安」的故事；② 从海外云**稳定可达**；③ demo 最后一天，实现成本要低；④ 免企业认证。

## 根因诊断（带证据）
加临时探测端点 `/debug/reach`，从 ECS 出口实测各推送服务真实 send 路径：
- `sctapi.ftqq.com` DNS 解析出**两个 IP**：`140.143.190.233` / `82.157.177.201`。
- Server酱真实 `…/{key}.send`：有时 3/3 秒回（400「错误的Key」≈450ms），有时 3/3 connect 超时。
- 结论：**两个 IP 里有一个从 AWS 新加坡出口被黑洞丢包**；而 Python `requests`/`urllib3` **只连 DNS 返回的第一个 IP，不像浏览器/curl 那样自动切到下一个**。撞到坏 IP 就超时、好 IP 就成 → 「时通时不通」。6/20 能通是走运，今天连环撞坏 IP。
- pushplus 同为国内小服务，同类风险；企业微信 `qyapi.weixin.qq.com`（腾讯，多 IP/CDN）探测 POST 351ms 干净返回 `errcode 93000`，稳定可达。

## 候选方案
1. **保留 Server酱/pushplus（个人微信）**：推到个人微信、零额外 App，但海外云不稳；要稳就得自己实现「多 IP 故障转移/重试」，脆弱且吃 25s 预算。
2. **企业微信群机器人 webhook（本次采用）**：腾讯大接口、海外云稳定，群机器人**免企业认证**、key 静态不过期。代价：落在「企业微信」App（微信家族，但非个人微信）。
3. **Telegram / Bark**：海外云几乎必通、实现最简，但**不是微信**，国内门卫场景不真实。
4. **后端迁香港/国内 region 或加代理**：能保个人微信，但最后一天动基础设施成本高、收益不匹配。

## 最终选择
方案 2：企业微信群机器人作为**云上主推送通道**，Server酱/pushplus 保留为本地/国内出口可用的备选。

实现：
- `config.py` 加 `WECOM_WEBHOOK_KEY`；`NOTIFIER_PROVIDER` 默认仍 `serverchan`（本地友好），云上由 CD 注入 `wecom`。
- `notifier.py` 加 `_wecom_send`：POST `qyapi.weixin.qq.com/cgi-bin/webhook/send?key=…`，`msgtype=text`；**显式判 `errcode!=0` 抛 `NotifyError`**（群机器人鉴权失败/限频会回 HTTP 200 + 非 0 码，不判会假成功）。
- `cd.yml`：`NOTIFIER_PROVIDER` 改为 `${{ vars.NOTIFIER_PROVIDER || 'wecom' }}`，注入 `WECOM_WEBHOOK_KEY` secret（沿用「只注入非空、不清空已有变量」的拼装逻辑）。
- 通道靠 `NOTIFIER_PROVIDER` 一键切换，正是当初做 notifier 抽象的价值兑现。

## 理由
- 直接消除「云→国内小服务被 IP 黑洞」这一不可控因素，换成腾讯大接口的稳定可达。
- 仍属微信家族，且工厂门卫本就常用企业微信，反而更贴近真实。
- 改动小：只加一个 provider + 一个 env，对话/登记/数据层零改动。

## 已知代价 / 可演进
- 通知落在企业微信 App，不是个人微信；演示需装企业微信。
- 群机器人无消息回执，发送成功仅代表「进群」，不代表保安已读（与 Server酱 一致，单向推送本就如此）。
- 群机器人有频率限制（20 条/分钟），登记量级远低于此，不影响。
- 可演进：生产可换企业微信自建应用（带应用消息/已读回执）、或后端迁国内 region 恢复个人微信通道。
