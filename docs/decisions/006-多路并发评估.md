# 006 多路并发评估与最小改造

## 背景 / 约束
- 多路并发是进阶方向之一：多通电话/多个登记请求可能同时到达。
- 当前 demo 使用 FastAPI + SQLite + Server酱。主链路已跑通，不应为了并发直接引入 Redis/Celery/PostgreSQL 等重组件。
- 25s 预算仍然重要；外部微信推送是慢 HTTP，不能让它阻塞整个事件循环。

## 当前风险
1. **async endpoint 内调用同步阻塞函数**：微信推送和 SQLite 都是同步调用，如果直接在 async 路由里跑，会阻塞 uvicorn event loop。
2. **SQLite 写锁**：SQLite 允许多读单写；并发写入时需要合理的 `busy_timeout`。
3. **重复工具调用竞争**：同一 `source_call_id` 并发到达时，先查再插可能产生竞态。
4. **外部推送慢/失败**：Server酱网络波动会影响登记响应耗时。

## 候选方案
1. **保持现状，只做说明**：最快，但无法证明并发能力。
2. **最小改造 + 压测验证**：SQLite 设置 `busy_timeout/WAL`；阻塞登记逻辑放线程池；增加 `request_id` 与耗时字段；用 `noop` 推送通道跑并发测试。
3. **生产化并发架构**：PostgreSQL + 队列 + worker + 可观测性系统。更稳，但明显超出 demo 阶段。

## 最终选择
方案 2：最小改造 + 压测验证。

## 实现要点
- SQLite：
  - 每个连接设置 `timeout`
  - `PRAGMA busy_timeout`
  - 初始化时启用 `WAL`，降低读写互相阻塞
  - 为 `source_call_id` 建唯一索引，避免并发重复登记
- FastAPI：
  - `/register_visitor` 与 `/vapi/register_visitor` 收到 JSON 后，把 `_register_core` 放进线程池执行，避免同步推送阻塞 event loop。
- 推送：
  - 新增 `NOTIFIER_PROVIDER=noop`，用于并发测试，不发真实微信。
- 可观测字段：
  - `request_id`
  - `elapsed_ms`
  - `push_elapsed_ms`

## 已知代价 / 可演进
- demo 阶段仍使用 SQLite，适合小规模并发，不等同于生产多门岗高并发。
- 微信推送仍是同步等待结果；生产可改成队列异步推送，并把登记成功与推送成功解耦。
- 生产化可演进为 PostgreSQL + async HTTP client + background worker + trace 系统。

## 验证结果（6/18）
测试配置：
- `DATABASE_PATH=concurrency_test.db`
- `NOTIFIER_PROVIDER=noop`，避免真实微信推送
- uvicorn 单进程，本地 HTTP 请求压测 `/register_visitor`

结果：
- 10 个唯一登记请求并发提交：`10/10 success`
- 总墙钟：约 `152ms`
- 单请求后端耗时：约 `52~137ms`
- `push_elapsed_ms=0ms`（noop 通道）
- 5 个相同 `source_call_id` 并发提交：`5/5 success/idempotent responses`
- 测试 DB 结果：总记录 `11` 条；相同 `source_call_id='same-call-id-race'` 只写入 `1` 条

结论：
- 当前 demo 级并发（5~10 个同时登记）下，FastAPI + SQLite + busy timeout/WAL 可以稳定通过。
- 唯一索引解决了同一通电话重复工具调用的并发竞态。
- 真实 Server酱推送仍可能成为慢点；生产化应改异步队列。
