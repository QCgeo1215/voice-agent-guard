# 002 凑齐后要不要复述确认

## 背景 / 约束
- 手机号 / 车牌"听错了也可能仍是合法值"（138→139、沪A12345→沪A12346），后端格式校验拦不住。
- 25s 是后端单次响应延迟预算，复述确认不影响它；复述影响的是对话轮次 / 体验。
- 体验目标：像真人门卫、别审问、轮次尽量少。

## 候选方案
1. **不额外确认**（原 prompt）：最快、轮次最少；但合法范围内的听错直接进库 / 推送。
2. **只确认手机号 + 车牌两项**：登记前念这两项请确认，对了再登记；多半轮~一轮，拦住最高风险字段；公司 / 事由不确认。
3. **确认全部 4 项**：最稳，但最啰嗦、轮次最多，和"像真人、别审问"冲突。

## 最终选择
方案 2：显式确认手机号 + 车牌两项。
- 用**显式**确认（念出来问"对吗"，等回应再调工具），不用隐式确认。原因：电话语音里"边说边调工具"时序不可控，错误可能已推送，无法纠正。
- 公司 / 事由不单独确认（保安能当场看出 / 影响小）。
- 轮次目标从 2-3 放宽到 3-4。

## 理由
- 高风险数字字段值得用一轮换数据正确性。
- 只确认 2 项，控制啰嗦，不破坏"像真人"体验。

## 已知代价 / 可演进
- 每通多半轮~一轮。
- 确认动作靠 LLM 执行（prompt 约束），无法 100% 保证它确实复述了；后续可在后端 / 通话日志观察。
- 未来可做"隐式确认 + 可打断"，更顺更省轮次。
