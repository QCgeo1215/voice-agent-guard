# 005 公司名标准化 / 园区固定实体

> 更新（6/20）：本决策中的“prompt 放园区公司白名单做软引导”已被 **决策 009** 修正。
> 当前做法：公司目录迁入 SQLite `companies` / `company_aliases`，`vapi/system_prompt.md` 不再包含完整白名单，只保留“后端工具校验公司，不要猜”的规则。

## 背景 / 约束
- 实际园区内公司是固定集合，不应把 ASR/LLM 听错的自由文本直接写入数据库。
- 语音中“蓝色鲸鱼科技”可能被听成“蓝色金云科技”“蓝色经于科技”“蓝色鲸鱼公司”等。
- 入库和推送必须使用标准公司名；如果公司不存在，应让访客确认，而不是制造脏数据。

## 候选方案
1. **只靠 prompt 写别名**：实现最轻，但 LLM 不稳定，数据库仍可能出现脏公司名。
2. **只在后端标准化**：数据库干净、可测试；但 Vapi 登记前确认时可能仍念错。
3. **两层标准化**：prompt 放园区公司白名单做软引导，后端做权威标准化和兜底。
4. **embedding / 向量检索**：适合公司很多且别名不可枚举的生产场景；但普通文本 embedding 更偏语义相似，不一定擅长“鲸鱼/精于”这类 ASR 音近错字。
5. **Hybrid resolver：候选召回 + 受限 LLM 重排**：先用白名单/别名/字符相似度召回 top 候选，再让 LLM 只能在候选里判断 match/uncertain/unknown。

## 最终选择
方案 3 + 5：后端权威 + prompt 软引导；后端内部升级为 Hybrid resolver。

- 后端维护模拟园区公司白名单、常见别名和轻量候选召回。
- exact/alias/high-confidence fuzzy 直接返回，毫秒级。
- 默认使用 fast 模式：中间置信度返回候选确认，不调 LLM，保证登记链路低延迟。
- 可通过 `COMPANY_RESOLVE_USE_LLM=true` 开启 LLM 受限重排实验；只有模糊分数处在中间区间时才调用。
- LLM 只允许输出白名单候选之一、`uncertain` 或 `unknown`，禁止生成新公司。
- 公司解析 LLM 独立短超时（默认 1.8s）；超时/失败就降级为候选确认，不阻塞主登记链路。
- 登记前把 `company` 标准化为白名单中的标准名，再入库 / 推送。
- 查询时也标准化公司过滤条件，避免用脏公司名查不到。
- 未命中公司时返回中文提示，请访客确认园区内公司名称。
- ~~Vapi prompt 中列出公司白名单和常见说法，让对话阶段尽量确认标准公司名。~~
- 6/20 起改为：prompt 不暴露完整公司目录；后端数据层是公司标准化权威，工具返回候选 / 不存在时由 Agent 追问确认。

## 运行策略
1. **确定性优先**：标准名 / alias 命中直接返回，通常 0ms。
2. **高置信 fuzzy**：字符相似度高且 top1 与 top2 差距足够大，直接归一到标准名，通常 0ms。
3. **中间置信度**：默认返回候选确认（如“您说的是蓝色鲸鱼科技吗？”），不直接写库。
4. **明显未知**：返回“园区里没查到这个来访单位”，并列出可登记公司。
5. **LLM rerank**：默认关闭。需要实验时设置 `COMPANY_RESOLVE_USE_LLM=true`，只对中间候选做受限重排。

## 理由
- 标准实体必须由后端兜底，不能只依赖 LLM。
- 当前公司数量小，用“候选召回 + 受限 LLM 重排”比直接上 embedding 更轻、更可解释。
- ASR 错误更多是音近/字形近，不是语义相似；普通文本 embedding 不是最优第一步。
- LLM 不直接面对全量自由空间，只做 top 候选的 constrained rerank，降低幻觉风险。
- 这能直接改善查询统计质量：按公司聚合不会被“蓝色鲸鱼 / 蓝色金云 / 蓝色经于”拆散。

## 已知代价 / 可演进
- 白名单要维护；真实生产应接园区企业目录或后台管理。
- LLM 默认关闭，因为测试中真实网络超时可能超过配置 timeout；demo/生产登记链路优先保证低延迟。需要展示智能 rerank 时可显式打开。
- 默认 fast 模式回归测试：26 个正反例全通过，最大耗时约 1ms；边缘输入返回候选确认。
- 公司数量变大后，可升级为拼音匹配、声学/拼音 embedding 召回、向量索引 + LLM rerank，或专门实体解析服务。
