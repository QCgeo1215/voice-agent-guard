# 004 门卫查询入口与技术栈是否升级

## 背景 / 约束
- 题目加分项之一是门卫可以查询访客数据。
- 当前微信方案是 Server酱，适合后端向个人微信单向推送，不适合在微信里双向问答。
- 主登记链路已经跑通，剩余时间应优先做稳定、可解释的增强，避免为了“看起来高级”重构底层。

## 候选方案
1. **维持后端 API / CLI 查询**：继续使用 `/guard/query` 和 `ask_guard.py`。最稳，但演示感弱。
2. **新增轻量网页后台**：保安打开网页输入自然语言问题，页面调用 `/guard/query`。可交互、可演示，不影响主链路。
3. **切换双向消息平台**：企业微信 / 公众号 / 飞书 / Telegram Bot。体验像聊天，但认证、配置和平台切换成本高。
4. **给 Vapi 增加查询工具**：让语音助手同时负责登记和查询。演示强，但多工具编排可能干扰主登记链路。

## 最终选择
方案 2：新增轻量网页后台，同时保留 `/guard/query` API。

## 理由
- Server酱继续负责“登记后通知保安”，网页后台负责“保安主动查询”，职责边界清楚。
- 不升级 Vapi / SQLite / FastAPI / Server酱 等底层组件，避免把已验证主链路重新置于风险中。
- 查询 Agent 继续采用“LLM 抽结构化意图 + 参数化 SQL”，不做 text-to-SQL，保证 SQL 可控、可测试、低注入风险。

## 已知代价 / 可演进
- 网页后台不是微信内聊天，交互感弱于企业微信/公众号。
- 生产化可接入企业微信/公众号/飞书 Bot，让查询入口变成双向消息。
- 若未来要语音查询，可单独设计第二个 Vapi Assistant 或查询工具，避免与访客登记流程混淆。
