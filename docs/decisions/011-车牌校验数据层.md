# 011 车牌校验与省份归一（plate_registry）

## 背景 / 约束
- 6/20 真机测试抓到 badcase：访客车牌「苏E…」被 STT 反复听岔，复述确认环节来回纠了 ~3 轮。结果对（没写错单），但体验差。
- 根因：车牌是「省份字 + 字母 + 数字」的无意义串，STT 声学识别率天然低；尤其单字省份（苏/沪）和字母（E）容易飘。
- 已有兜底：对话里的复述确认（system prompt 第2步）能保证最终正确，但不能降低轮数。
- 提出方向：参照公司白名单，给车牌也加「数据层」校验，而不是继续往 prompt 堆规则。

## 关键差异：公司是闭集，车牌是开集
- 公司：园区企业事先已知、数量有限 → 白名单能当「准入闸门」+ 标准化锚点；公司目录是园区业务数据 → 入库（决策 009）。
- 车牌：访客绝大多数是第一次来的陌生车，无法预先枚举 → **不能**做「不在名单就拒绝」的闸门，否则挡掉所有新访客。
- 所以车牌数据层的职责不是「成员校验」，而是：格式校验 + 省份闭集校验 + 省份首字近音纠正。
- 省份简称是**国家标准常量**（31 个、全国通用、十年不变），不是园区业务数据 → 放代码常量、**不入库**；为它建表属于过度设计。这是与公司目录刻意不对称的地方：对齐的是「模块职责」，不是「底层存储」。

## 候选方案
1. **继续在 prompt 里加车牌规则 / 罗列省份**：零代码，但持续膨胀 prompt、把业务校验塞给 LLM、与后端不一致。否。
2. **STT 关键词增强（keyword/keyterm boosting）**：朋友建议的「语料库」。对公司名（有意义词）有效，对车牌数字/字母串收益很小，且 Vapi 不一定开放该参数。非主力。
3. **后端 plate_registry 模块（本次采用）**：新增 `backend/plate_registry.py`，省份闭集常量 + 近音纠错表 + 归一/校验；main.py 收编原内联正则。结构对标 `company_registry.py`。
4. **历史车牌模糊纠错**：拿 `visitors` 历史表做匹配，老车听岔自动校回。有价值但与现有回访查询（`lookup_visitor` 按车牌查）重叠，且有「把合法车牌误改成相似历史车牌」的风险 → 暂不做，列为可演进。

## 最终选择
方案 3。

实现：
- 新增 `backend/plate_registry.py`：
  - `PROVINCE_ABBR`：31 个省份简称闭集（常量）。
  - `_PROVINCE_CONFUSABLE`：非省份近音/形近字 → 省份简称（仅当首字不在闭集时才查；跨省同音如桂/贵、甘/赣因都是合法省份字、不进此表，无歧义）。
  - `clean_plate()`：尽力归一不抛错（NFKC 全角转半角、去空格/点/横杠、字母大写、首字近音纠正），供回访查询复用。
  - `normalize_plate()`：在 clean 基础上做严格校验（首字 ∈ 省份集；其后 = 城市字母码 + 4~6 位字母数字，覆盖标准 7 位与新能源 8 位车牌），非法抛 `InvalidPlateError`。
- `backend/main.py`：删除内联 `_PLATE_RE`；`_validate_fields` 改调 `plate_registry.normalize_plate`，非法返回 `SPEECH_BAD_PLATE`；`_normalize_plate` 改为委托 `clean_plate`，回访查询同样受益于省份纠正。
- 单测 `backend/tests/test_plate_registry.py`：合法/全角/新能源/近音纠正/各类非法/clean 不抛错；本地 22 测全绿。

## 理由
- 后端是车牌的权威校验防线，挡掉假车牌与 STT 听成非省份字的情况，避免脏车牌入库。
- 省份近音纠正能把听岔的首字自动校回，减少复述确认轮数（针对 badcase）。
- prompt 不增长；业务校验留在后端，答辩口径与公司目录一致：「LLM 负责对话，后端负责事实与校验」。
- 零新依赖、复用现有结构，风险低。

## 已知代价 / 可演进
- 新访客那串数字本身（声学开集）后端无法凭空纠错，仍依赖复述确认兜底——这是开集的物理限制，任何白名单都救不了。
- 近音表是人工维护的常用省份子集，不全；后续可换 `pypinyin` 按拼音匹配，覆盖全部同音字、去掉手工维护。
- 历史车牌模糊纠错（方案 4）未做，可作为回访体验增强。
- 后端改动需重新部署 AWS 才线上生效；本地与 CI 已验证。
