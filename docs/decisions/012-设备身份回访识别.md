# 012 设备身份：接通即识别回访（不接电话）

## 背景 / 约束
- 题目「回访加分」的理想态是：老访客一来，系统**接通瞬间就认出他**，不用重报信息。电话通道天然有这个能力——caller ID 在零输入时就给出身份键。
- 但本项目入口是**扫码 Web Call**（决策 008），不是电话。诉求：在不接电话、不损失 STT 音质的前提下，尽量逼近「接通即识别」。
- 关键事实先厘清：
  - 回访识别**本来就用车牌做钥匙**（决策 007，`lookup_visitor` → `find_latest_by_plate`），与手机号无关。
  - 电话相对 Web 唯一多给的，不是「知道你是谁」，而是把识别时机从「司机开口报车牌」（第1轮）提前到「接通」（第0轮）。
  - Web 通道**先天拿不到设备的真实手机号**（浏览器无此 API，隐私限制）；首次接触必须靠对话采集。
- 已否电话方案（决策 008 + `phone_number_analysis.md`）：+86 个人短期拿不到合规号；窄带 8kHz 电话音频会拉低中文车牌/公司 STT——为省一个最弱字段牺牲三个更重要字段，不划算。

## 核心思路
回访需要的是「接通时有一个稳定身份键去查历史」，不是手机号。电话用 caller ID；Web 里**访客自己手机浏览器的 localStorage** 就是等价的「设备身份」。
- 设备只需记住「这台手机上次登记的**车牌**」，下次扫码把它交给现有的 `lookup_visitor(plate)`，就复刻了「接通即识别」。
- 不需要新建设备表、不需要新映射，完全复用现有以车牌为键的回访管线。

## 候选方案
1. **真电话 caller ID**：最贴题，但 +86 合规 + 窄带 STT 降质 + 计费。已否（决策 008）。
2. **一人一码 / 一链**：首访后发专属链接（带车牌）。跨设备可用，但要访客自己保存复用，与「门口一张海报码」模式不符，体验差。否。
3. **localStorage 设备身份（本次采用）**：首访结束后把车牌记到访客手机本机；回访扫码时经 Vapi `assistantOverrides.variableValues` 把 `known_plate` 带给 Agent，接通即查、即识别。
4. **维持现状**：回访仍只能在第1轮（报车牌后）识别，拿不到「接通即识别」。作为优雅降级的兜底，不作为目标。

## 最终选择
方案 3，且**只在本机记车牌**（不记手机号）。

为什么只记车牌就够、且更对：
- 车牌一旦在接通时命中，手机号/公司/事由会从数据库那条历史记录里自动带出（prompt 第0步：司机说「一样」即复用历史字段登记）。所以手机号**不必**存本机。
- 车牌与手机号没有严格绑定关系；号码永远以后端记录为准，不靠设备猜。司机换号了走 prompt 现成的「不一样」分支补一项。

实现：
- **前端 `/call`**（`backend/main.py` `_render_call_html`）：
  - 读 `localStorage["vg_known_plate"]`；非空时 `vapi.start(id, { variableValues:{ known_plate }, firstMessage:"…还是上次那辆车吗…" })`，覆盖开场白避免回访还问车牌；为空则保持原开场（新访客零回归）。
  - 通话拿到 `call.id`，结束时 `GET /visit/by-call/{call.id}` 取回本次车牌写入 localStorage。
- **后端**：新增只读 `GET /visit/by-call/{call_id}` → `find_by_call_id` → 只回车牌（最低敏感度），查不到回 null。
- **Vapi 控制台**：① system prompt 加 `{{known_plate}}` 上下文块 + 第0步「本机车牌非空就接通即查」规则；② register 工具请求体加模板字段 `source_call_id={{call.id}}`（让 Web 通话也有幂等键 + 支撑车牌回查）。
- 验证：`/call` 渲染 200、`/visit/by-call` 未命中→null / 命中→车牌；`pytest` 23 绿。

## 已知代价 / 可演进
- **是「同一浏览器」身份，不是「同一 SIM/人」**：换手机、清站点数据、换浏览器、无痕 → 当作新司机，优雅退回「报车牌」流程，无回退损失。对「送货司机用自己手机」这一真实场景足够。
- **首访免不了**：本机无记忆，仍需对话采集——与电话一致（首通也得采集，caller ID 也只省号码）。
- **设备≈人是信任假设**：他人借机扫码会被错认，但司机一句「不是/第一次来」即走 prompt「不一样」分支纠正，风险低。
- **依赖控制台两处配置**：`{{known_plate}}` 与 `source_call_id={{call.id}}`。未配时 `variableValues` 为空、`by-call` 查不到 → 功能优雅失效、不报错。
- 可演进：把开场白做成「念出上次公司」（需 model-generated first message，代价是首条延迟 + 新访客开场也变动，暂不做）；或叠加门禁/IC 卡等更强身份。
