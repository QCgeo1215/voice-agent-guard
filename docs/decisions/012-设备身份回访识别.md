# 012 设备身份：接通即识别回访（不接电话）

## 背景 / 约束
- 回访的理想态是：老访客一来，系统**接通瞬间就认出他**，不用重报信息。电话通道天然有这个能力——caller ID 在零输入时就给出身份键。
- 但本项目入口是**扫码 Web Call**（决策 008），不是电话。诉求：在不接电话、不损失 STT 音质的前提下，尽量逼近「接通即识别」。
- 关键事实先厘清：
  - 回访识别（决策 007）既能按车牌也能按手机号查（`lookup_visitor`：先 `find_latest_by_plate`，未命中再 `find_latest_by_phone`）。
  - 电话相对 Web 唯一多给的，不是「知道你是谁」，而是把识别时机从「司机开口报车牌」（第1轮）提前到「接通」（第0轮）。
  - Web 通道**先天拿不到设备的真实手机号**（浏览器无此 API，隐私限制）；首次接触必须靠对话采集。
- 已否电话方案（决策 008 + `phone_number_analysis.md`）：+86 个人短期拿不到合规号；窄带 8kHz 电话音频会拉低中文车牌/公司 STT——为省一个最弱字段牺牲三个更重要字段，不划算。

## 核心思路
回访需要的是「接通时有一个稳定身份键去查历史」。电话用 caller ID（手机号）；Web 里**访客手机浏览器 localStorage 存的上次手机号**就是等价的「来电号码」——同一手机再扫码＝同一来电。
- 接通时按这个手机号查最近一次历史，把**车牌/公司/事由**复述给司机确认；**手机号本身全程不再问**（已知）。
- 完全复用现有 `find_latest_by_phone` + register 管线，不新建设备表。
- 本质：用「扫码」模拟「打电话」——电话接通即知的是手机号，车牌/公司/事由才是要现场确认的。

## 候选方案
1. **真电话 caller ID**：最贴题，但 +86 合规 + 窄带 STT 降质 + 计费。已否（决策 008）。
2. **一人一码 / 一链**：首访后发专属链接（带车牌）。跨设备可用，但要访客自己保存复用，与「门口一张海报码」模式不符，体验差。否。
3. **localStorage 设备身份（本次采用）**：首访结束后把**手机号**记到访客手机本机；回访扫码时 prefetch 查最近历史，经 Vapi `assistantOverrides.variableValues` 把 `known_phone` + 车牌/公司/事由带给 Agent，接通即识别。
4. **维持现状**：回访仍只能在第1轮（报车牌后）识别，拿不到「接通即识别」。作为优雅降级的兜底，不作为目标。

## 最终选择
方案 3，**本机记手机号**（caller ID 等价物），不记车牌。

为什么记手机号：
- 在用「扫码」模拟「打电话」，电话接通即知的就是手机号，所以本机身份键＝手机号最贴合直觉。
- 手机号命中后按它取最近一次历史，把车牌/公司/事由复述确认；手机号不再追问。司机说「有更新」走 prompt 分支只改那一项。
- 早期曾记车牌（已废弃）：会出现「假设车牌、反而不复述车牌」的别扭，与电话直觉相反；现以手机号为键纠正。

实现：
- **前端 `/call`**（`backend/main.py` `_render_call_html`）：
  - 读 `localStorage["vg_known_phone"]`；非空时 prefetch `POST /lookup_visitor {phone}` 取最近历史，再 `vapi.start(id,{ variableValues:{ known_phone, known_plate, known_company, known_reason }, firstMessage: 复述句 })`；为空保持原开场（新访客零回归）。
  - 通话拿到 `call.id`，结束时 `GET /visit/by-call/{call.id}` 取回本次**手机号**写入 localStorage。
- **后端**：`GET /visit/by-call/{call_id}` 返回本次 `phone`（车牌一并回传仅兼容/调试），查不到回 null；`_lookup_core` 返回补 `plate_number` 供复述。
- **Vapi 控制台**：① system prompt 用 `{{known_phone}}`（已知不再问）+ `{{known_plate/company/reason}}`（复述确认）；② register 工具体加模板字段 `source_call_id={{call.id}}`（Web 通话幂等键 + 支撑手机号回查）。
- 验证：phone 身份链路本地冒烟通过（按手机号查带回车牌、`by-call` 回手机号）。

## 已知代价 / 可演进
- **是「同一浏览器」身份，不是「同一 SIM/人」**：换手机、清站点数据、换浏览器、无痕 → 当作新号/新设备，优雅退回完整采集流程，无回退损失。对「送货司机用自己手机」这一真实场景足够。
- **首访免不了**：本机无记忆，仍需对话采集——与电话一致（首通也得采集，caller ID 也只省号码）。
- **设备≈人是信任假设**：他人借机扫码会被错认，但司机说「有更新/不一样」即走 prompt 分支纠正，风险低。
- **存的是手机号、比车牌更敏感**：但只落在访客自己手机本机、且仅其本人可见，演示场景可接受；后端记录仍是唯一权威。
- **依赖控制台两处配置**：`{{known_phone}}`（+ `{{known_plate/company/reason}}`）与 `source_call_id={{call.id}}`。未配时 `variableValues` 为空、`by-call` 查不到 → 功能优雅失效、不报错。
- 可演进：把开场白做成「念出上次公司」（需 model-generated first message，代价是首条延迟 + 新访客开场也变动，暂不做）；或叠加门禁/IC 卡等更强身份。
