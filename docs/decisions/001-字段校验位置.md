# 001 字段校验逻辑放哪

## 背景 / 约束
- 语音场景 STT 对数字（手机号 / 车牌）易听错，错误数据会进库并推给保安。
- 25s 延迟预算，不希望为校验多绕工具往返。
- 个人独立开发、demo 需稳定。

## 候选方案
1. **只放后端**：单一可信源、规则集中可测；但要等一次工具往返才发现错误，多一轮 + 延迟，后端话术经 LLM 转述可能走样。
2. **只放 prompt**：发现早、对话自然、零后端改动；但赌 LLM 可靠性，规则散在自然语言里难维护，换模型表现不稳。
3. **两层（后端为主 + prompt 软拦）**：后端权威兜底 + prompt 当场软拦减少往返；代价是同一套规则两处重复。

## 最终选择
方案 3，后端为主、prompt 为辅，校验从宽。
- 后端（权威）：手机号 `^1\d{10}$`（归一化去分隔符后）；车牌 `^[一个汉字][A-Za-z0-9]{5,7}$`，**不**强制省份白名单，避免近音误杀新能源 / 特殊车牌；失败返回 `success:false` + 中文重说话术。
- prompt（软拦）：要求 LLM 收集后先快速核一遍，明显不对就请重说一项；并处理后端"重说"回退，更新后重新登记。

## 理由
- 数据正确性必须有后端最终防线，不能只赌 LLM。
- 纯后端在语音里多绕一轮体验差，prompt 软拦减少往返。
- 校验从宽：宁可漏放个别畸形输入，也不误杀正常车牌（误杀代价 = 多问一次，但正常流程不能被卡）。

## 已知代价 / 可演进
- 同一套规则后端与 prompt 各写一份，有重复，以后端为准。
- 车牌仅"结构校验"，未做省份白名单与校验位算法。
- 中文数字（"幺三五…"）暂不在后端转换，依赖 STT / LLM 输出阿拉伯数字。
