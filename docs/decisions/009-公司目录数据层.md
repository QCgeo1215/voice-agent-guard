# 009 公司目录数据层（移出 Prompt）

## 背景 / 约束
- 6/19 手机端实测后暴露出一个设计问题：`vapi/system_prompt.md` 里直接包含完整公司白名单和别名。
- 这在 demo 初期能提升 LLM 理解稳定性，但不适合继续保留：
  - prompt 和 `backend/company_registry.py` 有两份公司目录，容易不同步。
  - 公司目录属于业务配置，不应暴露给 LLM prompt。
  - 真实园区企业目录会变化，不应该靠改 prompt / Vapi 控制台维护。
  - LLM 应负责对话编排，事实校验和标准化应由后端负责。

## 候选方案
1. **继续 prompt + 后端双份维护**：最省事，但重复来源、泄露配置、维护差。
2. **JSON 配置文件**：把名单从 prompt 移到 `data/companies.json`，后端读取。简单，但仍不是业务数据库。
3. **SQLite 公司目录表（本次采用）**：新增 `companies` / `company_aliases` 表，启动时从 seed 导入；后端从表读取并缓存。贴合当前 SQLite 技术栈，也更接近生产。
4. **远程企业目录 / 后台维护**：生产最佳，但当前无真实目录和管理后台需求。

## 最终选择
方案 3：SQLite 公司目录表 + seed 数据。

实现：
- 新增 `backend/data/companies.json` 作为 demo seed。
- `db.init_db()` 新建：
  - `companies`
  - `company_aliases`
- 当 `companies` 为空时，从 seed 导入；已有数据不覆盖，便于后续人工维护。
- `company_registry.py` 改为从 `db.load_company_registry()` 读取公司目录并缓存。
- `vapi/system_prompt.md` 删除完整公司名单，只保留规则：公司目录由后端工具校验，LLM 不要猜，按工具 message 追问。

## 理由
- 后端成为唯一事实来源，避免 prompt 与代码/数据不同步。
- 公司目录不再暴露给 LLM prompt，答辩时更好解释“LLM 负责对话，后端负责事实与校验”。
- 复用现有 SQLite，不引入额外组件，风险低。
- seed 数据仍可版本管理；未来可以替换为后台维护或真实企业目录。

## 已知代价 / 可演进
- 当前 seed 仍在仓库里，只是不再在 prompt 中暴露；真实生产应接企业目录或后台维护。
- `company_registry.py` 对公司目录做进程内缓存；如果运行时修改公司表，需要调用 `reload_company_registry()` 或重启服务。
- 后续可做：
  - 公司目录管理页面
  - 拼音/声学匹配
  - 园区企业目录 API 同步
  - PostgreSQL 持久化（配合云部署）
