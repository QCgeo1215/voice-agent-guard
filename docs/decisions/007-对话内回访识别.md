# 007 对话内回访识别（lookup 工具，升级 003）

## 背景 / 约束
- 把回访的**理想态**写成可衡量的正例：开口就主动确认、跳过重复采集（"张师傅您好，今天是不是和上周二一样来蓝色鲸鱼送货？"，2 轮 ~8s）。
- 决策 003 当时选了方案 3「事后欢迎回来」（正常采集 4 项 + 登记时标注），明确否决了对话内 lookup（方案 2），理由：多一次工具往返吃延迟、LLM 编排两个工具不稳、为一个可选增强性价比低。
- **新信息**：复盘需求后，把"匹配回访理想态"列为优先级。这是推翻 003 的依据（决策可被新信息推翻）。
- demo 是 Web Call，**没有主叫号**；对话里最早稳定拿到的回访键是**车牌**（司机通常一开口就报）。

## 候选方案
1. 维持 003 方案 3：不动。最省事，但到不了理想态（仍问全 4 项、无主动确认）。
2. **对话内 lookup（本次采用）**：新增 `lookup_visitor` 工具，Agent 拿到车牌后先查；命中则主动确认"上次一样吗"，确认后复用历史 company/reason/phone 直接登记。命中路径 ~2 轮。
3. 主叫号识别（生产最优）：接通即认出。demo 无电话线路，做不到，留演进。

## 最终选择
方案 2。
- 后端：新增 `db.find_latest_by_plate`（精确匹配）+ `count_by_plate`；`_lookup_core` 先按车牌精确查、未命中再按手机号兜底，返回 `found / visit_count / company / reason / phone / last_date / message`。
- 端点：`POST /lookup_visitor`（扁平主用，对齐 register 的方案 A）+ `/vapi/lookup_visitor`（信封备用，result 为紧凑 JSON 串）。
- prompt：新增"第 0 步 识别回访"——报完车牌即调 lookup；命中念 message 确认，司机说"一样"就用历史值跳到登记。

## 复述确认（决策 002）的再取舍
- **回访命中且确认"一样"：跳过复述，直接登记。** 依据：车牌是查询命中键（已匹配）、手机号取自历史（上次已确认），两个高风险字段都可信，002 的复述对回访冗余。
- **新访客：保留决策 002**（仍显式确认手机号 + 车牌）。准确性优先；回访路径已足够快，两者各取所需。

## 已知代价 / 可演进
- 新访客多一次 lookup 工具往返（本地查库几 ms，但多一个 LLM 回合）；命中回访才有正收益。这是 003 当时规避的成本，现按回访优先级接受。
- 回访键以车牌为主：同车牌换司机、或司机换车，会误判 / 漏判；手机号兜底缓解换车场景。
- 依赖 LLM 正确编排两个工具（先 lookup 后 register）；prompt 已用严格分步约束，实拨需复测稳定性。
- 演进：有真实电话线路后上主叫号识别（方案 3），实现"接通即认出"。
