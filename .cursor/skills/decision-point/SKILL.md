# 决策点 (Decision Point)

## 何时触发
遇到以下情况，先停下来走本流程，不要直接写代码：
- 技术选型（平台、库、模型、协议）
- 系统架构 / 数据流 / 状态管理的结构性选择
- 有明显 trade-off 的取舍（灵活 vs 稳定、延迟 vs 准确、成本 vs 体验）
- 一处改动会影响多个模块或后续演进方向

## 流程
1. **约束与目标**：一句话讲清这个决策要解决什么、硬约束是什么（25s 延迟预算、demo 稳定、个人可独立完成等）。
2. **候选方案**：列 2-3 个，每个写「做法一句话 + 优点 + 缺点/风险」。至少 2 个，不要只给一个。
3. **推荐**：给出倾向和理由，对齐到约束。
4. **讨论**：和用户确认或调整后再实现。关键分歧点摆出来，别替用户拍板。
5. **实现**：达成一致才写代码。
6. **记录**：在 `docs/decisions/` 写一条，作为答辩「思考过程」材料。

## 决策记录格式（docs/decisions/NNN-标题.md）
- 背景 / 约束
- 候选方案（含被否决的，写明为什么否）
- 最终选择 + 理由
- 已知代价 / 后续可演进点

## 原则
- 考察的是技术判断，不是写代码速度；架构选择 > 代码量。
- 没有银弹，每个选择都说清放弃了什么。
- 决策可被推翻：有新信息就回来更新记录。
